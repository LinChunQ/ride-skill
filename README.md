# RIDE Flow - 结构化开发工作流

一个通用的 AI 编程助手工作流 skill，通过四个顺序阶段引导功能实现，并自动管理项目文档。

## 概述

RIDE Flow 强制执行规范的功能开发方法：

- **R** - Requirement Alignment (需求对齐) - 编码前澄清所有细节
- **I** - Implementation Confirmed (确认实施) - 创建全面的设计文档
- **D** - Development (开发) - 编码的同时维护活文档
- **E** - Export Docs / Endpoint Handoff (导出/对接文档) - 生成前端可用的规范文档

## 自动文档管理

当你启动此工作流时，系统会自动：

1. 在 `docs/` 目录下创建以需求名称命名的项目子目录（例如：`docs/用户认证/`）
2. 在项目目录中创建三个文档文件：
   - `需求名称-说明.md` - 记录最终确认的需求说明
   - `需求名称-实施方案.md` - 记录架构设计、数据模型、代码结构和变更历史
   - `需求名称-API对接文档.md` - 记录前端可直接使用的 API 规范

**目录结构示例**：
```
docs/
└── 用户认证/
    ├── 用户认证-需求说明.md
    ├── 用户认证-实施方案.md
    └── 用户认证-API对接文档.md
```

## 安装

根据你使用的 AI 编程工具：

- **支持 skill 的 AI 工具**: 复制到对应的 skills 目录（通常为 `~/.{工具名}/skills/ride/`）
- **其他工具**: 参考对应工具的 skill 安装文档

重启或重新加载 skills 后即可使用。

## 发布新版本 (Release)

本项目已配置 GitHub Actions 全自动打包工作流。当修改完核心逻辑（如 `SKILL.md`）需要发布新版本时，请严格按照以下命令执行，系统会自动在 GitHub Releases 页面生成供 CC Switch 或其他工具直接下载的 ZIP 安装包：

```bash
# 1. 提交本地的代码变更到主分支
git add .
git commit -m "feat: 描述你这次更新了什么功能"
git push

# 2. 为本次发布打上版本号 Tag（⚠️ 注意：Tag 名称必须以 v 开头）
git tag v1.0.1

# 3. 推送 Tag 到远端，这一步会触发自动打包发布流程
git push --tags
```

## 使用方法

### 方式 1: 在支持 skill 的工具中（如 Claude Code）

安装后使用命令启动：
```
/ride 需求名称
```

例如：
```
/ride 用户认证
```

### 方式 2: 在不支持 skill 的工具中（如 Codex、Cursor 等）

直接对 AI 说：
```
使用 RIDE 工作流来实现用户登录功能
```

或者：
```
启动 RIDE 流程，需求是：添加基于 JWT 的用户认证
```

或者：
```
按照 RIDE 开发流程来做订单管理模块
```

AI 会自动识别并启动 RIDE 工作流。

## 工作流阶段

### 阶段 R: 需求对齐 (Requirement Alignment)
- 不编写任何代码
- AI 助手会针对实施细节提出 3-5 个澄清问题
- 关注并发、数据模型、错误处理、缓存、边界情况
- 需求说明持续更新到 `需求名称-说明.md`
- **文档保持简洁**：每次更新直接覆盖整个文档，只保留最终的完整需求说明，不保留历史对话记录
- 仅在需求完全理解后才推进

### 阶段 I: 确认实施 (Implementation Confirmed)
- 在 `需求名称-实施方案.md` 中创建详细的实施方案
- 包含架构图（Mermaid）、数据模型、代码结构、API 契约
- 需要用户批准后才能继续

### 阶段 D: 开发 (Development)
- 根据方案实施功能
- 当发生变更时自动更新 `需求名称-实施方案.md`
- 在文档末尾的"变更记录"章节记录所有偏离
- 保持代码与文档 100% 同步

### 阶段 E: 导出/对接文档 (Export Docs / Endpoint Handoff)
- 由用户请求触发
- 扫描 Controllers/Routers 和 DTOs
- 将 API 规范写入 `需求名称-API对接文档.md`
- 产出 AI 可解析、前端可直接使用的 API 规范

## 优势

- **防止范围蔓延** - 编码前锁定需求
- **活文档** - 文档与代码保持同步
- **前端交接** - 零歧义的 API 规范给前端团队
- **结构化思考** - 强制提前考虑边界情况
- **中文友好** - 所有文档、注释均使用中文
- **自动管理** - 每个需求自动创建独立的文档目录，使用清晰的命名格式
- **变更追踪** - 记录所有偏离原始方案的变更及原因
- **文档简洁** - 需求文档只保留最终版本，避免冗余的历史记录

## 示例

```
用户: /ride 用户认证

AI: [初始化完成]
已创建项目目录：docs/用户认证/
已创建文档文件：
- 用户认证-说明.md
- 用户认证-实施方案.md
- 用户认证-API对接文档.md

[阶段 R] 请说明或粘贴你的初始需求。

用户: 添加基于 JWT 的用户认证

AI: 我将帮你实现 JWT 认证。让我先澄清一些细节：

1. Token 存储：refresh token 应该存储在数据库还是 Redis？
2. Token 过期时间：access token 和 refresh token 的期望生命周期是多久？
3. 并发会话：是否允许用户同时拥有多个活跃会话？
4. 密码要求：是否有特定的验证规则？
5. 限流：是否需要实现登录尝试次数限制？

[根据你的回答，需求说明文档会持续更新，只保留最终的完整需求描述]
```
